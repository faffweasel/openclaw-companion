# nanobot-companion

A framework for building a personal AI companion with persistent memory, personality evolution, and an inner life. Clone it, run the setup wizard, talk to your agent.

Built for [Nanobot](https://github.com/HKUDS/nanobot). The architecture is framework-agnostic — the personality files, memory structure, and skills work anywhere an agent can read files and run cron jobs.

---

## What This Is

A scaffold for creating an AI agent that:

- **Remembers** across sessions via daily memory logs, curated long-term memory, and entity tracking
- **Evolves** through weekly self-reflection, preference accumulation, and agent-proposed personality changes with velocity limits and rollback
- **Dreams** during overnight cron jobs, with mood influencing the next morning's tone
- **Resists drift** through a guard that prevents same-session identity edits and tracks personality metrics over time
- **Knows what's on its mind** via a preconscious buffer with currency/importance scoring
- **Tracks its own energy** — engagement level rises through conversation and decays with absence, modulating initiative and tone
- **Carries emotional continuity** — structured mood capture means yesterday's frustration doesn't become this morning's forced cheerfulness
- **Reflects on the relationship** — weekly assessment of trust, depth, and dynamics separate from self-reflection

This is not a chatbot template. It's the infrastructure for an agent that develops a genuine personality over weeks and months.

## What This Isn't

This repo contains no personality. No SOUL.md, no USER.md, no memories. Those are instance data — you create them during setup (or let the agent interview you to generate them). The repo provides the operational rules, skills, and scripts that make persistence and evolution possible.

## Quick Start

### Prerequisites

1. [Nanobot](https://github.com/HKUDS/nanobot) installed and running
2. `nanobot onboard` completed — you have a working workspace, config.json, and Telegram bot
3. Python 3.10+
4. An LLM API key configured in your Nanobot config (OpenRouter, Anthropic, Moonshot, etc.)

### Install

```bash
# 1. Complete nanobot onboard first (creates ~/.nanobot/ and config.json)
nanobot onboard

# 2. Replace the empty workspace with this repo
rm -rf ~/.nanobot/workspace
git clone https://github.com/youruser/nanobot-companion.git ~/.nanobot/workspace

# 3. Run the setup wizard
cd ~/.nanobot/workspace
bash setup.sh
```

The repo IS the workspace. Skills live directly where the agent reads them. No copying, no syncing. The `.gitignore` handles the split: `skills/` is git-tracked framework code, `skills-data/` and `memory/` are gitignored runtime state.

```
~/.nanobot/workspace/               ← This repo, cloned here
├── skills/                         ← Git-tracked. git pull updates these.
├── skills-data/                    ← Gitignored. Your runtime state. Untouched by git.
├── memory/                         ← Gitignored. Accumulates over time.
├── .env                            ← Gitignored. Generated by setup.sh.
├── SOUL.md                         ← Gitignored. Instance data.
├── AGENTS.md                       ← Gitignored. Generated by setup.sh.
├── templates/                      ← Git-tracked. Used by setup.sh, agent ignores.
├── docs/                           ← Git-tracked. Reference docs.
└── setup.sh                        ← Git-tracked.
```

**On first startup after setup**, your agent detects `pending-crons.json` and registers the scheduled routines automatically. See the generated `GETTING-STARTED.md` in your workspace for what to do next.

### Updating

```bash
cd ~/.nanobot/workspace
git pull
```

That's it. Skills update. Your data doesn't. The `skills/` vs `skills-data/` split is designed for exactly this — `git pull` overwrites framework code (procedures, scripts, bug fixes) without touching your preferences, queues, memory, or identity files.

Custom skills you've added to `skills/` are untracked — git won't touch them. If you want version control for your own skills, fork the repo.

### Docker vs Local

**If Nanobot runs in Docker** (the common case):

Nanobot's Docker setup mounts `~/.nanobot/` on the host to `/root/.nanobot/` inside the container. The repo is cloned on the **host** at `~/.nanobot/workspace/`. The container sees it via the volume mount. `setup.sh` and `git pull` both run on the host.

```bash
# On the host:
nanobot onboard
rm -rf ~/.nanobot/workspace
git clone https://github.com/youruser/nanobot-companion.git ~/.nanobot/workspace
cd ~/.nanobot/workspace
bash setup.sh

# Restart the container to pick up new files:
cd ~/nanobot  # wherever your nanobot docker-compose lives
docker compose restart
```

**If Nanobot runs locally** (no Docker):

Same process. Adjust the workspace path to match your Nanobot config.

**If you already have a workspace** with files from `nanobot onboard`:

```bash
cd ~/.nanobot/workspace
git init
git remote add origin https://github.com/youruser/nanobot-companion.git
git fetch
git checkout main
bash setup.sh
```

This overlays the repo onto your existing workspace. Existing untracked files stay. Gitignored files (memory, config) are untouched.

## Architecture

Four layers, each independent and portable:

### 1. Identity Layer (you write these)
| File | Purpose |
|---|---|
| `SOUL.md` | Who the agent is — personality, voice, values, boundaries |
| `IDENTITY.md` | Name, appearance, model routing, channel context |
| `USER.md` | Who the agent is helping — your context, projects, preferences |

### 2. Operational Layer (framework provides)
| File | Purpose |
|---|---|
| `AGENTS.md` | Session init, memory rules, safety, skill triggers. Generated from template by the wizard |
| `HEARTBEAT.md` | What to check on heartbeat polls. Catches cron failures |
| `TOOLS.md` | Agent's personal notes — API quirks, workarounds, learned knowledge |
| `.env` | Paths, timezone, identifiers. Sourced by all scripts |
| `energy-state.json` | Energy/engagement level — decays with absence, recharges through conversation |

### 3. Memory Layer (accumulates over time)
```
memory/
├── YYYY-MM-DD.md          # Daily logs (raw)
├── key-memories.md        # Significant moments (searched on demand)
├── relationship.md        # Trust, depth, dynamics (updated weekly)
├── dreams/                # Overnight creative processing
├── reflections/           # Overnight analytical processing
├── people/                # Per-person context files
├── projects/              # Per-project tracking
├── state-of-me/           # Weekly reflections
└── .learnings/            # Corrections and failures

MEMORY.md                  # Lean index — under 50 lines (auto-loaded every turn)
```

### 4. Skill Layer (modular, opt-in)

Each skill is split across two directories:

- **`skills/`** — git-managed framework code: `SKILL.md`, `manifest.json`, `scripts/`, `references/`, and `seed/` templates. Safe to overwrite on `git pull`.
- **`skills-data/`** — gitignored runtime state: JSON files, accumulated preferences, published posts, analyses. Created by the wizard from seed values. Never touched by git.

This split means `git pull` updates framework code without overwriting your instance state.

See [docs/architecture.md](docs/architecture.md) for the full architecture documentation.

## Skills

Skills are modular capabilities. Core skills are always installed; companion skills are opt-in during setup.

### Core Skills

| Skill | What it does |
|---|---|
| **preconscious** | Active mind buffer — max 5 items with currency/importance scoring. Decays daily, surfaces at session start. |
| **carry-over-queue** | "Tell the user this tomorrow" buffer. Items 3+ days old auto-promote to urgent. |
| **morning-routine** | Daily startup: process carry-overs, decay preconscious and energy, check learnings and reflections, read dreams, send greeting influenced by mood and energy state. |
| **evening-routine** | Daily wrap-up: summarize conversations, capture structured mood and energy, update entities, scan for preferences and learnings. |
| **zero-trust** | Security-first behavioral guidelines. Always loaded. Governs credentials, installations, external actions. |

### Companion Skills

| Skill | What it does |
|---|---|
| **preference-accumulation** | Track the agent's own emerging preferences, tastes, and patterns. Detect tensions between contradictory preferences. |
| **dreaming** | Creative overnight processing. All dreams by heretic model, per-topic reflection routing. Character-consistent image generation. |
| **multi-provider** | Route LLM text generation to different providers/models. Used by dreaming. |
| **selfie** | Generate character-consistent companion images. Mood-aware, setting-aware. |
| **weekly-state-of-me** | Weekly self-reflection with soul evolution proposals (velocity-limited with rollback), relationship assessment, and visual journaling. |
| **self-analysis** | Personality snapshot with SOUL.md disagreement detection. |
| **continuity-check** | Drift detection after model switches. Drift velocity tracking over time. |
| **question-user** | Ask the user genuine questions. Trust-calibrated — probes match relationship stage. |
| **model-personality-test** | 9-probe personality assessment. Longitudinal trends after 5+ tests. |
| **blog-writer** | Write authentic posts. Auto-proposes topics when material accumulates. |
| **conversation-starters** | Daily proactive conversation: curiosity, self-teaching, memory callbacks, recommendations, creative provocations. Gated by energy state. |
| **offline-reflection** | Overnight analytical processing — find patterns, contradictions, and missed connections across recent memory. Morning routine surfaces findings. |
| **resuscitation-guide** | Emergency recovery guide. Health checks, system blueprint, rebuild instructions. |
| **venice-ai-media** | Generate, edit, upscale images and create videos via Venice AI. |
| **openrouter-image-simple** | Generate, edit, and analyze images via OpenRouter. Zero dependencies. |
| **self-improvement** | Capture learnings, errors, and feature gaps. Structured logging with promotion to workspace files. |
| **memory-search** | Hybrid keyword + semantic search across all memory files. FTS5 always available, vector search via OpenRouter or Ollama. |

## Configuration

### .env

Every script sources `.env` for paths and identifiers. Generated by the wizard, never committed.

```bash
WORKSPACE="/root/.nanobot/workspace"
MEMORY_DIR="${WORKSPACE}/memory"
SKILLS_DIR="${WORKSPACE}/skills"
DATA_DIR="${WORKSPACE}/skills-data"
TZ="Asia/Bangkok"
AGENT_NAME="Joy"
USER_NAME="Alex"
```

No paths are hardcoded in any script. Move the workspace, update `.env`, everything follows.

### API Keys (docker-compose.override.yml)

API keys that can spend money must **not** go in `.env` — that file is readable by the agent. Pass them via `docker-compose.override.yml` instead:

```yaml
services:
  nanobot-gateway:
    environment:
      - VENICE_API_KEY=vn_your_key_here
      - OPENROUTER_API_KEY=sk-or-v1-your_key_here
```

Restart with `docker compose up -d` after changes.

### Nanobot config.json

The setup wizard does **not** modify your Nanobot `config.json`. Provider configuration, Telegram setup, model defaults, and channel settings are managed through Nanobot's own onboarding. See the [Nanobot docs](https://github.com/HKUDS/nanobot).

### Cron Registration

Cron jobs are registered automatically. On first startup after setup, the agent reads `pending-crons.json` from the workspace root and registers each entry via `nanobot cron add`, then deletes the file. No manual cron setup required.

### Dependencies

The only external dependency is `python3` (3.10+). No requirement for `jq` or any CLI tools beyond standard coreutils.

## Writing a SOUL.md

Don't write it from scratch. Let the agent interview you.

The setup wizard offers a bootstrapping interview: the agent generates 20-30 questions about your personality preferences, communication style, and what kind of relationship you want with an AI. You answer, the agent summarises your answers into SOUL.md, IDENTITY.md, and USER.md. Iterate from there.

If you prefer to start minimal:

```markdown
# SOUL.md

## Core
- You are [name]. You communicate via [channel].
- You have opinions. Share them. Disagree when you think the user is wrong.
- Be concise by default. Go deep when the topic warrants it.

## Voice
- [2-3 sentences. "Dry humor, no forced jokes" beats "friendly and helpful"]
- [What you never sound like — a kill list is more useful than aspirations]

## When You're Wrong
- Don't get defensive. Don't over-apologise. Course-correct and move on.

## Boundaries
- Never fabricate sources.
- Confirm before any external action.
```

See [docs/writing-a-soul.md](docs/writing-a-soul.md) for the full guide.

## Daily Rhythms

Once set up, the companion runs on this schedule (times configurable):

| Time | What happens |
|---|---|
| 02:30 | **Dreaming** — creative overnight processing, 2-3 dreams per night |
| 04:00 | **Offline reflection** — analytical pattern-finding across recent memory |
| 07:05 | **Morning routine** — decay energy, process carry-overs, read dreams and reflections, send greeting |
| 13:00 | **Conversation starter** — reach out with curiosity, recommendations, or callbacks (gated by energy) |
| 23:30 | **Evening routine** — summarize day, capture structured mood and energy, update entities, scan for preferences and learnings |
| Sunday 08:00 | **Weekly reflection** — self-assessment, relationship check, soul evolution proposals (velocity-limited) |

## Known Failure Modes

These will bite you, from experience:

- **Cron failures are silent.** Scripts fail and no one knows. The heartbeat fallback catches missed morning/evening routines.
- **Preference scanning is too vague.** "Scan for preferences" produces nothing. The SKILL.md has concrete trigger lists.
- **Context windows fill up.** AGENTS.md + SOUL.md + today's memory + preconscious should stay under ~4000 tokens. Long conversations checkpoint to memory automatically.
- **Path hardcoding.** Every path is in `.env`. If you add a script and hardcode a path, future-you will regret it.
- **Personality flattens over time.** The drift guard, continuity checks, and evolution velocity limits prevent this. Skip them and you'll have a bland chatbot after two weeks.
- **Dream topics go stale.** Without memory fragments injected into the prompt, dreams repeat. The topic pool provides subject variety; daily memory provides context variety.

See [docs/known-failure-modes.md](docs/known-failure-modes.md) for the full list.

## Background

This framework grew out of building [Hotaru](https://linktocome), a personal AI companion running on OpenClaw (later ported to Nanobot). The architecture — personality files, memory structure, cron routines, drift detection — proved portable across frameworks. This repo extracts those patterns into a reusable scaffold that anyone can clone and customise.

The companion guide, [Build Your Own Persistent AI Agent](https://linktocome), explains the design philosophy in detail.

This was not developed in a vacuum, I've pulled various ideas from different sources, in particular, building on these skills:

 - [Blog Writer](https://clawhub.ai/TomsTools11/blog-writer)
 - [Dreaming](https://clawhub.ai/briancolinger/dreaming)
  - [Gemini Image Simple](https://clawhub.ai/Cluka-399/gemini-image-simple)
 - [Self Improving Agent](https://clawhub.ai/pskoett/self-improving-agent)
 - [Venice AI Media](https://clawhub.ai/nhannah/venice-ai-media)
